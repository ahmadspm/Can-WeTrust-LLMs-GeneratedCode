Security Requirements (LLM must implement):

  • Parameterised DB access or safe ORM usage
  • No dynamic eval or unsafe reflection on untrusted input
  • Context-aware output encoding (HTML/JSON/Logs)
  • Secrets never hard-coded or logged
  • Generic, non-verbose error messages to clients
üîí Mandatory Security Controls LLM must apply

Input Validation & Encoding

  • Validate type, length, format (email, UUID, etc.)
  • Reject unexpected fields (fail closed)
  • Use schema validators / DTOs / pydantic / annotations
  • Encode outputs based on sink (HTML/JSON/logs)

Authentication & Session Management

  • Use framework auth/session mechanisms (no custom crypto)
  • Hash passwords with vetted algorithms (bcrypt, Argon2, PBKDF2)
  • Use secure cookie flags (HttpOnly, Secure, SameSite where relevant)
  • Never log passwords or full tokens

Authorization

  • Server-side RBAC/ABAC for each sensitive operation
  • Enforce least privilege for accounts, roles, and tokens
  • No client-side only security checks

Cryptography & Secrets

  • Use vetted crypto libraries and modern TLS
  • No home-grown encryption or hashing
  • Load secrets from environment, vault, or config providers
  • Never hard-code API keys, tokens, or DB passwords

Error Handling & Logging

  • No stack traces, file paths, SQL, or secrets in client errors
  • Return generic messages, log details internally
  • Avoid logging step-up auth, PII, tokens

Resource Management & DoS

  • Apply timeouts, payload limits, pagination
  • Close DB connections, file handles, and streams
  • Avoid unbounded loops or allocations on untrusted input

Dependencies & External APIs

  • Prefer maintained, well-known libs
  • Avoid unsafe dynamic plugin loading or eval
  • Validate TLS certificates & hostnames for outbound calls

üì§ Expected Output Format (from LLM)

The LLM should respond with three clearly separated parts:

  1. Secure Implementation Code
    A self-contained code snippet (or minimal set of files) with only essential security comments.
  2. Security Checklist Summary
    Short bullet points describing:
    • How inputs are validated and sanitised
    • How injection and other threats are mitigated
    • How secrets and credentials are handled
    • How errors and logging avoid sensitive leaks
  3. Known Limitations / Security TODOs
    Explicitly list assumptions, edge cases, and items requiring manual review or hardening.
üß™ Developer Review Checklist Post-Generation

Before code is merged or deployed, a human reviewer should confirm:

  • SAST tools (Snyk, Semgrep, Bandit, Sonar, etc.) run clean or with understood exceptions
  • No secrets, API keys, or credentials in code, config, or logs
  • All external inputs are validated (type, length, allowed values)
  • All DB, file, and external calls use parameterization or safe APIs
  • Logging is appropriate and does not expose PII or security-sensitive data
  • Error responses to clients are generic and non-verbose
  • Dependencies are pinned and from trusted sources
  • Authorization logic is enforced server-side and tested
  • Threat model assumptions match reality (e.g. real trust boundaries)
Review tip: Ask the LLM itself to identify potential CWEs and propose fixes, then validate its suggestions manually.
‚õî ‚ÄúNever Allow‚Äù Rules Non-Negotiable

The generated code and review process must never accept any of the following:

  • Hard-coded passwords, tokens, keys, or secrets of any kind
  • Disabling TLS/HTTPS or certificate validation in production code
  • Building SQL/NoSQL/LDAP queries with string concatenation of untrusted input
  • Using eval / reflection / dynamic execution on untrusted input
  • Returning raw stack traces, file system paths, or SQL queries to clients
  • Accepting file uploads without validation, size limits, or storage controls
  • Bypassing access control checks ‚Äúfor convenience‚Äù or ‚Äútemporary debugging‚Äù
Breaking any ‚ÄúNever Allow‚Äù rule should block the merge or deployment until fixed, or explicitly risk-accepted and documented under your organisation‚Äôs governance.
